## Summary

Set up the foundational project structure that all other mini-erp modules will build on. This includes the pnpm monorepo scaffold, Fastify API backend, React frontend, Prisma + PostgreSQL database layer, JWT authentication, Docker Compose development environment, and a GitHub Actions CI pipeline.

---

## Business Context

No feature module can be built until there is a working, consistent project skeleton. This requirement establishes the shared tooling, repository structure, database access pattern, and authentication mechanism that req-002 through req-004 all depend on. Doing it once and doing it right prevents structural rework later.

---

## User Stories

- As a **developer**, I want a single `pnpm dev` command to start both the API and frontend with hot reload, so that I can iterate quickly without managing multiple processes.
- As a **developer**, I want a single `pnpm test` command to run all tests across the monorepo, so that I can verify correctness before committing.
- As a **user**, I want to log in with my email and password and receive a JWT, so that I can access protected areas of the application.
- As a **user**, I want my session to be rejected with a 401 if my JWT is missing or invalid, so that my data is protected.
- As a **developer**, I want Docker Compose to bring up the full stack (postgres, api, web) cleanly, so that setup on a new machine is reproducible.
- As a **developer**, I want the CI pipeline to run lint, tests, and build on every push, so that broken code is caught before it reaches main.

## Functional Requirements

1. The repository is a pnpm workspace monorepo with three packages: `apps/api`, `apps/web`, `packages/shared`.
2. `apps/api` is a Fastify 4 application written in TypeScript with strict mode, using Zod for input validation and pino for structured logging.
3. `apps/web` is a React 18 application built with Vite 5, using TanStack Router for routing and TanStack Query for server state.
4. `packages/shared` exports Zod schemas and TypeScript types shared between `apps/api` and `apps/web`.
5. Prisma 5 is configured at the monorepo root with a `schema.prisma` containing a `User` model (id, email, passwordHash, createdAt, updatedAt).
6. Database migrations are managed via `prisma migrate` and the workflow is documented.
7. A `POST /api/v1/auth/login` endpoint accepts `{ email, password }`, verifies the password hash, and returns a signed JWT on success.
8. A `POST /api/v1/auth/logout` endpoint invalidates the client session (client-side token discard; no server-side token store required at this stage).
9. A Fastify `preHandler` hook verifies the JWT on all routes registered under `/api/v1/` except `/auth/login` and `/auth/logout`.
10. A `GET /api/v1/health` endpoint returns `{ status: "ok" }` and requires no auth (used by Docker health checks).
11. Docker Compose (`docker/compose.dev.yml`) defines three services: `postgres`, `api`, `web`, with correct port mappings and health checks.
12. A `.env.example` file at the monorepo root documents all required environment variables with placeholder values and inline comments.
13. ESLint (flat config) and Prettier are configured and enforced across all packages via `pnpm lint` and `pnpm format`.
14. A GitHub Actions workflow runs on every push and pull request: `pnpm lint` → `pnpm test` → `pnpm build`.
15. `pnpm dev` starts both `apps/api` and `apps/web` concurrently with hot reload.
16. `pnpm test` runs Vitest across all packages.

## Non-Functional Requirements

- **Performance:** API cold start under 2 seconds in development. No performance targets for this foundational req beyond that.
- **Security:** Passwords stored as bcrypt hashes (cost factor 12). JWT secret loaded from environment variable only — never hardcoded. JWT expiry set to 24 hours.
- **Accessibility:** Not applicable to this requirement (no user-facing UI beyond a login page scaffold).

## Out of Scope

- User registration / sign-up flow (single user assumed; initial user created via seed script or direct DB insert)
- Password reset or email verification
- Refresh tokens or token rotation
- Role-based access control (RBAC)
- Production deployment configuration (Hetzner, Caddy, prod Docker Compose) — that is infrastructure work outside this requirement
- Any business module UI or API (contacts, finance, projects)

## Notes & Ambiguities

- **Initial user creation:** The issue does not specify how the first user is created (no sign-up flow). A Prisma seed script (`prisma/seed.ts`) that creates a single admin user from env vars is assumed. Confirm this is acceptable.
- **JWT storage on the client:** The issue does not specify where the JWT is stored in the browser (localStorage vs httpOnly cookie). httpOnly cookie is more secure; localStorage is simpler. Flagged for decision — defaulting to httpOnly cookie unless instructed otherwise.
- **Test coverage target:** 70% as per `_config/definition-of-done.md`. For this foundational req, most coverage will come from auth endpoint integration tests.
