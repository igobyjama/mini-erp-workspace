# Naming Conventions

## Branch Names

| Type | Pattern | Example |
|------|---------|---------|
| Feature | `feature/req-NNN-short-description` | `feature/req-001-project-foundation` |
| Bug fix | `fix/req-NNN-short-description` | `fix/req-003-invoice-total-rounding` |
| Hotfix | `hotfix/req-NNN-short-description` | `hotfix/req-004-task-delete-crash` |
| Release | `release/vX.Y.Z` | `release/v1.0.0` |

Rules:
- Always lowercase, words separated by hyphens
- Always include `req-NNN` so branches trace back to requirements
- Keep the description short (3–5 words max)

---

## Commit Messages

Format: `type(req-NNN): short description`

| Type | When to use |
|------|-------------|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation only |
| `test` | Adding or updating tests |
| `refactor` | Code change that is not a feature or fix |
| `chore` | Build process, tooling, dependencies |

Example: `feat(req-002): add contact CRUD endpoints`

Rules:
- Subject line max 72 characters
- Use imperative mood ("add", not "added" or "adds")
- Reference issue in PR description with `Closes #NNN`, not in every commit

## File Names

| Context | Convention | Example |
|---------|-----------|---------|
| React components | `PascalCase.tsx` | `ContactList.tsx` |
| React hooks | `useCamelCase.ts` | `useContacts.ts` |
| Fastify route files | `camelCase.routes.ts` | `contacts.routes.ts` |
| Service files | `camelCase.service.ts` | `contacts.service.ts` |
| Zod schema files | `camelCase.schema.ts` | `contact.schema.ts` |
| Prisma migrations | auto-generated by Prisma CLI | |
| Test files | `*.test.ts` / `*.test.tsx` | `contacts.service.test.ts` |
| Docker files | `Dockerfile.<env>` | `Dockerfile.prod` |

---

## Database (Prisma)

- Model names: PascalCase singular (`Contact`, `Project`, `Task`)
- Field names: camelCase (`firstName`, `createdAt`)
- Enum names: PascalCase (`ContactType`, `TaskStatus`)
- Enum values: SCREAMING_SNAKE_CASE (`CLIENT`, `VENDOR`, `IN_PROGRESS`)
- Many-to-many join tables: auto-managed by Prisma where possible

## API Endpoints

Pattern: `/api/v1/<resource>[/:id][/<sub-resource>]`

| Method | Path | Action |
|--------|------|--------|
| GET | `/api/v1/contacts` | List all contacts |
| POST | `/api/v1/contacts` | Create contact |
| GET | `/api/v1/contacts/:id` | Get single contact |
| PATCH | `/api/v1/contacts/:id` | Update contact |
| DELETE | `/api/v1/contacts/:id` | Delete contact |
| GET | `/api/v1/projects/:id/tasks` | List tasks for a project |

Rules:
- Resources are lowercase plural nouns
- Use PATCH (not PUT) for partial updates
- IDs are UUIDs (Prisma `@id @default(uuid())`)

---

## Pull Request Titles

Format: `[req-NNN] Short description of the change`

Example: `[req-002] Add contacts module (clients & vendors)`

Rules:
- Always include `[req-NNN]`
- Always include `Closes #NNN` in the PR body to auto-close the GitHub issue on merge
